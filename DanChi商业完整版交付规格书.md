# DanChi 商业完整版交付规格书

版本：v1.0  
目标质量：8.5-9.0 分商业级 Android 英语背单词 App  
首发方向：Android 英语背单词  
首发核心词库：中考必备  
适用对象：AI 编程代理、开发团队、产品验收、测试验收、内容团队  

---

## 1. 目标定义

本规格书不是 MVP 文档，而是面向完整商业 App 的交付标准。目标不是“能跑”，而是“可长期自用、可小规模商用、可上架、可运营、可迭代”。

目标评分：

| 评分 | 含义 |
| --- | --- |
| 5 分 | 能运行的 Demo，有核心页面但体验粗糙 |
| 6 分 | 可自用原型，学习流程可用但内容、稳定性、测试不足 |
| 7 分 | 可内测版本，核心流程稳定，有基本内容和统计 |
| 8 分 | 可公测版本，有完整学习闭环、较好 UI、内容审核、基础后台 |
| 8.5 分 | 可商业候选版本，具备上架、付费、同步、监控、灰度、反馈能力 |
| 9 分 | 接近成熟商业产品，内容质量、体验细节、稳定性、运营工具齐备 |

本项目目标最低为 8.5 分，理想目标为 9 分。

---

## 2. 关键判断

直接把产品文档交给 AI 一次性实现，通常只能得到 5-6 分版本。要达到 8.5-9 分，必须把任务拆成多个可验收阶段，并且每个阶段都要求：

- 能编译。
- 能真机运行。
- 有自动化测试。
- 有内容样例。
- 有错误处理。
- 有状态恢复。
- 有验收截图或录屏。
- 有明确不合格标准。

AI 可以完成：

- Android 工程搭建。
- 页面和导航。
- 本地数据库。
- 复习算法。
- 学习流程。
- 词典查询。
- 统计页面。
- 后台 API 初版。
- 管理后台初版。
- 自动化测试初版。

AI 不能替代：

- 真实可商用词库授权。
- 人工内容审核。
- 高质量真人发音授权。
- 真机体验打磨。
- 用户访谈。
- 上架合规最终判断。
- 付费转化验证。

### 3.2 明确不做

即使是完整版，第一商业版本也不做：

- 日语、韩语。
- iOS。
- 微信小程序。
- 大规模社交广场。
- 排行榜 PK。
- 未审核的用户公开内容社区。
- 盗用第三方词典内容。
- 未授权真人发音。
- 强制广告换学习资格。

## 5. 工程架构要求

### 5.0 技术选型最终决策

根据“简单易用、服务器资源占用低、技术方向尽量现代、便于 AI 分阶段实现”的要求，首版商业候选版本采用以下方案：

| 层 | 选型 | 说明 |
| --- | --- | --- |
| Android 客户端 | Kotlin + Jetpack Compose + Room | 原生体验、离线、音频、通知更稳 |
| 后端核心 | PocketBase + Go 扩展 | 单文件部署，内置 SQLite、Auth、文件、Admin Dashboard、REST API，资源占用低 |
| 后台管理 | PocketBase Admin 为主，少量自定义管理页为辅 | 内容、反馈、用户、订单、配置先用内置后台集合管理 |
| 数据库 | SQLite 起步，定期备份 | 首版低成本；数据增长后再评估 PostgreSQL/独立服务 |
| 短信登录 | 手机号 + 短信验证码 | 通过阿里云短信或腾讯云短信接入，封装供应商接口 |
| 支付 | 微信支付 + 支付宝 App 支付 | 订单创建、签名、回调验签必须在服务端完成 |
| 内容包 | JSONL/ZIP + manifest + checksum | App 可下载、校验、导入、回滚 |
| 监控 | 客户端崩溃监控 + 服务端日志 | 初期轻量实现，后续接专业 APM |

为什么不首选 Directus：

- Directus 后台能力更强，但 Node 服务、数据库和部署复杂度更高。
- 当前项目优先低资源和快速可控，PocketBase 的内置后台已经覆盖首版大部分管理需求。
- 如果后期内容团队扩大、审核流复杂，再把内容后台迁移到 Directus 或独立 CMS。

为什么不首选 Bun/Hono：

- Bun + Hono 很现代、轻量、适合自研 API，但后台管理需要额外开发。
- 当前更缺的是“简单可用的后台 + 低资源后端”，PocketBase 更符合首版商业候选版本。
- 后期如果需要边缘部署或高并发 API，可以引入 Hono/Bun 做独立网关或内容服务。

PocketBase 风险说明：

- PocketBase 官方文档说明其仍处于 v1.0 之前，生产关键应用需要关注版本变更和迁移。
- 因此首版必须做好数据库备份、内容包备份和导出机制。
- 如果用户规模、支付订单量或内容审核流程明显增长，应评估迁移到 PostgreSQL + 独立 API + 独立后台。

### 5.1 Android 工程模块

推荐多模块结构：

```text
DanChi/
  settings.gradle.kts
  build.gradle.kts
  app/
  core-common/
  core-domain/
  core-data/
  core-database/
  core-audio/
  core-network/
  core-analytics/
  core-design/
  feature-home/
  feature-study/
  feature-dictionary/
  feature-wordbook/
  feature-statistics/
  feature-profile/
  feature-settings/
  feature-auth/
  feature-subscription/
```

最低分层要求：

- `feature-*` 不直接访问 Room DAO。
- `feature-*` 只通过 UseCase 或 Repository 获取数据。
- `core-domain` 不依赖 Android Context。
- `core-domain` 不依赖 Compose。
- `core-data` 负责协调本地数据库、网络和同步。
- `core-audio` 统一处理音频播放和 TTS。
- `core-analytics` 只暴露业务事件接口。
- `core-design` 统一颜色、字体、间距、组件。

### 5.3 后端架构

商用候选版本需要后端，不能只有本地 App。

首版后端以 PocketBase 为核心，采用“内置集合 + 自定义 Go 路由 + 定时任务”的结构。

```text
backend/
  main.go                  PocketBase 启动和扩展入口
  internal/
    auth/                  手机号验证码登录
    sms/                   阿里云/腾讯云短信适配
    sync/                  学习记录同步
    content/               内容包上传、发布、回滚
    payment/               微信/支付宝订单与回调
    entitlement/           会员权益计算
    config/                远程配置
    feedback/              内容纠错和意见反馈
    backup/                SQLite 备份和内容包备份
  pb_migrations/           PocketBase collection 迁移
  pb_hooks/                可选 JS hooks
  storage/                 内容包、音频、备份
```

最低接口：

```text
POST /auth/login
POST /auth/refresh
GET  /me

GET  /content/wordbooks
GET  /content/wordbooks/{id}/manifest
GET  /content/packages/{version}

POST /sync/review-logs
GET  /sync/review-states
POST /sync/favorites
POST /sync/notes

POST /feedback/content-report
POST /feedback/app-feedback

GET  /config/app
GET  /subscription/entitlements
POST /payment/orders
POST /payment/verify
```

PocketBase 集合至少包括：

- `users`：用户基础信息，手机号唯一。
- `sms_codes`：验证码记录，带过期时间和使用状态。
- `wordbooks`：词库元信息。
- `content_packages`：内容包版本、checksum、发布状态。
- `review_logs`：云端学习记录。
- `review_states`：云端复习状态。
- `favorites`：收藏。
- `notes`：笔记。
- `study_plans`：学习计划。
- `feedbacks`：意见反馈。
- `content_reports`：内容纠错。
- `orders`：订单。
- `payments`：支付流水。
- `entitlements`：会员权益。
- `remote_configs`：远程配置。

### 5.4 手机号登录要求

账号方式采用手机号验证码登录。

必须实现：

- 发送验证码接口。
- 校验验证码接口。
- 验证码有效期，建议 5 分钟。
- 单手机号发送频控。
- 单 IP 发送频控。
- 图形验证码或风控预留。
- 游客账号绑定手机号。
- 换绑手机号。
- 注销账号。

短信供应商：

- 首选阿里云短信或腾讯云短信。
- 供应商调用封装为接口，避免业务代码绑定单一服务商。
- 短信签名和模板需按服务商要求审核。

### 5.5 支付要求

支付方式采用微信支付和支付宝 App 支付。

必须遵守：

- App 只发起支付请求，不生成最终支付签名。
- 订单创建、签名、验签、回调处理必须在服务端完成。
- 支付成功以服务端回调和主动查询结果为准，不能只相信客户端回调。
- 微信支付和支付宝的密钥、证书、私钥不得放在 App 端。
- 支付回调必须幂等。
- 订单状态必须可查询。
- 支付失败、取消、超时、退款、权益恢复都要有状态。

订单状态：

| 状态 | 说明 |
| --- | --- |
| created | 已创建 |
| pending | 等待支付 |
| paid | 已支付 |
| failed | 支付失败 |
| cancelled | 用户取消 |
| expired | 超时关闭 |
| refunded | 已退款 |
| closed | 已关闭 |

权益处理：

- 支付成功后创建或延长 `entitlements`。
- App 启动和会员页都要能刷新权益。
- 用户换设备后可恢复权益。
- 退款后权益按规则取消或降级。

## 6. 数据库交付要求

### 6.1 必须建表

本地数据库至少包含：

- `words`
- `word_meanings`
- `example_sentences`
- `word_books`
- `word_book_items`
- `review_states`
- `review_logs`
- `favorite_words`
- `user_notes`
- `study_plans`
- `audio_assets`
- `content_sources`
- `content_packages`
- `sync_queue`
- `user_profile`

### 6.2 必须支持

- 数据库版本迁移。
- 内容包版本管理。
- 学习记录增量同步。
- 离线写入同步队列。
- 收藏和笔记离线编辑。
- 学习计划本地保存。
- 内容删除或替换后的兼容处理。

### 6.3 关键约束

- `word_id` 必须稳定，不随内容包更新变化。
- `review_state` 以用户-学习项为维度，不应只依赖页面状态。
- `review_log` 只追加，不随便覆盖。
- `content_package` 和 `review_log` 分离，更新内容不能删除用户学习记录。
- 笔记和收藏必须支持后续云同步。

## 7. 内容交付要求

### 7.1 中考必备词库

商业候选版本要求：

- 1,600-1,700 个中考核心词。
- 每个词有稳定 ID。
- 每个词至少 1 个核心释义。
- 每个词至少 1 条例句。
- 每个词有词性。
- 每个词有英美发音字段。
- 每个词有内容来源记录。
- 每个词有审核状态。

### 7.3 内容审核

每个词条状态：

| 状态 | 说明 |
| --- | --- |
| draft | 初稿 |
| reviewed_basic | 拼写、词性、释义已审 |
| reviewed_exam | 中考适配已审 |
| reviewed_legal | 来源和版权已审 |
| published | 已发布 |
| deprecated | 已下线 |

商业候选版不允许 `draft` 词条进入正式内容包。

### 7.4 词库来源和第三方词典边界

用户希望参考的词典包括：

- 有道词典。
- 牛津词典。
- 金山词霸。
- 柯林斯词典。
- 剑桥词典。
- 韦氏词典。

这些词典可作为“权威参考对象”和“未来授权/API 接入对象”，但在没有正式授权或明确 API 许可前，不得作为批量抓取、缓存、内置释义、内置例句、内置音频、内置品牌 Logo 的数据来源。

即使当前阶段“不商用”，也不建议把未授权词典内容导入本地词库，原因：

- 后续商业化时很难彻底清理来源不明的释义、例句和音频。
- 应用上架、合作、融资或售卖内容包时会遇到版权审计。
- 未授权抓取可能违反网站条款、版权规则或反爬限制。
- 词库一旦进入用户学习数据，后续替换成本很高。

### 7.4.1 哪些词库数据需要授权

不是所有“单词数据”都需要商业授权。应按数据类型分级判断。

| 数据类型 | 是否通常需要商业授权 | 说明 |
| --- | --- | --- |
| 英文单词本身，如 `apple`、`important` | 通常不需要 | 单个词、短语、事实性信息通常不是版权保护重点 |
| 自己整理的“中考必备 1650 词”清单 | 不一定需要 | 如果基于公开标准、公开教材要求、自己筛选排序形成，通常可作为自有编排；不要复制竞品独家词表顺序和分类 |
| 官方课程标准中的要求和公开考试范围 | 看来源和使用方式 | 可用于参考和归纳，不应整篇复制受保护材料 |
| ECDICT、WordNet 等开放许可数据 | 按许可证使用 | 可商用与否取决于许可证；必须保留版权声明和 attribution 要求 |
| Wiktionary / CC BY-SA 数据 | 可商用但有条件 | 通常要求署名和相同方式共享，可能影响内容包授权方式 |
| 有道/牛津/金山/柯林斯/剑桥/韦氏的释义 | 需要授权或 API 许可 | 释义是词典核心内容，不能批量复制内置 |
| 这些词典的例句、短语、搭配、同反义词编排 | 需要授权或 API 许可 | 属于编辑加工和表达内容 |
| 这些词典的真人音频、发音文件 | 需要授权 | 音频是独立作品或邻接权内容 |
| 这些词典的品牌 Logo、词典名背书 | 需要授权或遵守品牌规则 | 未授权不要宣称“内置牛津/柯林斯”等 |
| 自己写的中文释义、自己写的例句、自己生成并审核的 TTS 音频 | 不需要第三方授权 | 仍需做好内部内容审核和来源记录 |

结论：

- “词表”不等于“词典内容”。
- 我们可以自己做中考词表和自有释义例句。
- 不能无授权复制商业词典的释义、例句、音频和品牌内容。
- 商业授权主要针对“受保护的词典内容”和“数据库/API 使用权”，不是针对英语单词本身。

允许的研发方式：

| 方式 | 是否允许 | 说明 |
| --- | --- | --- |
| 官方 API 试用 | 允许 | 必须遵守 API 条款、调用量、缓存限制 |
| 外部跳转查询 | 允许 | 打开浏览器或 WebView，不缓存对方内容 |
| 手工少量核对 | 允许 | 只用于人工校对，不把原文复制进内容包 |
| 批量爬取释义 | 不允许 | 未授权时不得执行 |
| 批量下载音频 | 不允许 | 未授权时不得执行 |
| 使用词典 Logo 宣称内置 | 不允许 | 未授权不得使用品牌资产 |
| 用户个人笔记 | 谨慎允许 | 用户自己输入，默认仅个人可见，不作为官方内容 |

### 7.5 可执行的首版词库方案

首版中考必备词库建议采用：

1. 公开课程标准和考试词频作为词表候选。
2. ECDICT、WordNet、Wiktionary 等许可证明确的数据作为基础参考。
3. AI 生成初版中考释义和例句。
4. 人工审核并重写为自有表达。
5. TTS 生成英美音频兜底。
6. 用户纠错回流修订。

内容审核时应记录：

- 候选词来源。
- 释义参考来源。
- 是否人工改写。
- 例句是否 AI 生成。
- 例句是否人工审核。
- 音频来源。
- 是否允许商用。

### 7.5.1 AI 生成内容策略

DanChi 可以使用 AI 生成以下官方内容的初稿：

- 中文核心释义。
- 中考常用义排序。
- 英文例句。
- 中文例句翻译。
- 搭配。
- 短语。
- 同义词。
- 反义词。
- 易混词。
- 词根词缀解释。
- 谐音/联想助记。
- 考点提示。
- 拼写易错点。

但 AI 输出不得直接无审核发布。必须经过自动校验和人工审核。

AI 生成原则：

- 提示词中不得粘贴或要求复刻有道、牛津、金山、柯林斯、剑桥、韦氏等词典原文。
- 不要求 AI “模仿某词典风格”。
- 释义要短、清楚、适合中考，不追求词典大而全。
- 例句要符合初中语境，避免成人化、暴力、低俗、歧视、政治敏感内容。
- 例句尽量使用中考范围内词汇。
- 同反义词和搭配必须标记置信度，低置信度不进正式内容包。
- 谐音助记不得误导发音，不使用低俗联想。

### 7.5.2 AI 内容自动校验

AI 生成后必须经过自动校验：

| 校验项 | 规则 |
| --- | --- |
| 拼写 | 单词拼写必须和词表一致 |
| 词性 | 词性必须在允许枚举内 |
| 释义长度 | 中考核心释义优先，避免长段解释 |
| 例句长度 | 初中阶段建议 6-16 个英文词 |
| 例句目标词 | 例句必须包含目标词或合理变形 |
| 超纲词检测 | 例句中超纲词过多则退回 |
| 敏感内容 | 命中敏感词或不适合 K12 内容则退回 |
| 重复度 | 与已有例句过度相似则退回 |
| 版权相似度 | 与第三方词典例句/释义高度相似则退回 |
| 同反义词 | 必须能在开放数据或规则库中交叉验证 |
| 搭配短语 | 必须符合常见英语表达，不生成生造搭配 |
| 中文翻译 | 不得明显漏译或错译 |

相似度检查：

- 对有授权或开放许可的参考数据做相似度检测。
- 对用户上传或人工录入的第三方词典参考材料只做内部冲突检查，不进入内容包。
- 高度相似内容必须重写或删除。

### 7.5.3 人工审核分级

不是所有 AI 内容都需要同等强度人工审核。建议分级：

| 内容 | 审核强度 |
| --- | --- |
| 核心释义 | 必须人工审核 |
| 例句和翻译 | 必须人工审核 |
| 中考核心词搭配/短语 | 必须人工审核 |
| 同反义词 | 抽检 + 高频词必审 |
| 词根词缀 | 必须人工审核 |
| 谐音/联想助记 | 必须人工审核 |
| 低频扩展内容 | 自动校验 + 抽检 |

审核状态：

- `ai_generated`
- `auto_checked`
- `human_review_required`
- `human_reviewed`
- `rejected`
- `published`

正式内容包只允许 `human_reviewed` 或明确允许抽检发布的内容进入。

### 7.5.4 TTS 音频策略

音频可以直接使用 TTS，不需要使用第三方词典真人音频。

推荐策略：

- MVP/内测：Android 系统 TTS 实时播放作为兜底。
- 商业候选版：使用可商用 TTS 服务预生成高频词英美音频和例句音频。
- App 本地缓存音频文件，提高播放稳定性。
- 音频 manifest 记录 voice、accent、provider、license、生成时间、checksum。
- 用户可切换美式/英式发音。

TTS 注意事项：

- 必须确认 TTS 服务允许商业使用和 App 内分发。
- 不使用“克隆某真人声音”的音色。
- 不使用让用户误以为是真人词典音频的文案。
- 音频文件若由 AI 合成，应按适用法规和平台要求保留生成来源和必要标识。
- TTS 生成失败时，App 端回退系统 TTS。

### 7.5.5 AI 生成合规记录

每条 AI 生成内容应记录：

| 字段 | 说明 |
| --- | --- |
| generation_model | 使用的模型 |
| prompt_version | 提示词版本 |
| generated_at | 生成时间 |
| reviewer | 审核人 |
| review_status | 审核状态 |
| source_refs | 参考来源，不存未授权原文 |
| similarity_score | 与参考内容相似度 |
| risk_flags | 敏感、超纲、相似、低置信度等标记 |

如果未来在 App 内提供“用户点击生成例句/讲解”的实时 AI 功能，则需要额外考虑生成式 AI 服务合规、内容安全审核和 AI 生成内容标识。本阶段官方词库内容属于内部生产流程，仍建议后台保留 AI 生成记录和审核状态。

## 8. 学习算法要求

### 8.1 评分按钮

用户按钮：

- 忘记
- 模糊
- 认识
- 熟练

算法映射：

- 忘记 -> Again

- 熟练 -> Easy

### 8.2 必须支持

- 新词学习队列。
- 到期复习队列。
- 逾期复习处理。

- 听写错误影响掌握度。
- 熟词暂停和恢复。

### 8.3 测试要求

必须有测试覆盖：

- 新词首次评分后生成下次复习时间。
- Again 会进入更短间隔。
- Easy 会进入更长间隔。
- 到期词会进入今日复习。
- 超过复习上限时队列按优先级截断。
- 用户暂停熟词后不再进入日常队列。
- 恢复熟词后能重新进入复习。

## 13. AI 实施方式

### 13.1 不允许一次性全量生成

即使用 GPT-5.5 High，也不建议一次性让 AI 写完整 App。必须按阶段执行，每阶段独立验收。

推荐阶段：

1. 工程骨架和设计系统。
2. 数据库和内容包导入。
3. 首页、词库、学习计划。
4. 新学和复习流程。
5. 拼写、听写、释义选择。
6. 词典、收藏、笔记。
7. 统计、设置、提醒。
8. 账号、同步、后端接口。
9. 会员、支付、权益恢复。
10. 后台管理。
11. 监控、埋点、远程配置。
12. 测试、真机修复、上架材料。

每阶段必须：

- 编译通过。
- 运行通过。
- 提交测试结果。
- 说明已知问题。
- 不破坏上一阶段功能。

### 13.2 AI 代码约束

必须遵守：

- 不引入 GPL/AGPL 代码。
- 不抓取竞品词库。
- 不写死大段假数据在 UI 中。
- 不把业务逻辑写进 Compose 页面。
- 不跳过错误状态。
- 不跳过数据库迁移。
- 不用未授权词典 Logo。
- 不直接内置有道、牛津、金山、柯林斯、剑桥、韦氏内容。
- 不把支付成功只做成本地开关。
- 不把同步做成覆盖式上传导致数据丢失。

### 13.3 每阶段交付格式

AI 每阶段必须输出：

- 修改文件列表。
- 新增模块说明。
- 关键实现说明。
- 测试命令。
- 测试结果。
- 未完成项。
- 风险。
- 下一阶段建议。

### 13.4 推荐给 AI 的总提示词

可以将以下提示词和两份文档一起交给 AI：

```text
你是负责实现 DanChi 商业级 Android 英语背单词 App 的高级工程代理。

目标不是 MVP，而是 8.5-9 分商业候选版本。你必须严格遵守：
1. 以《背单词App开发文档.md》作为产品总纲。
2. 以《DanChi商业完整版交付规格书.md》作为工程交付和验收标准。
3. 首发只做 Android 英语中考必备，不做 iOS、小程序、日语、韩语。
4. 不允许引入 GPL/AGPL 代码。
5. 不允许复制有道、牛津、金山、柯林斯、剑桥、韦氏或竞品内容。
6. 不允许一次性生成完整大工程，必须按阶段实现，每阶段可编译、可运行、可测试。
7. UI、Domain、Data、Audio、Analytics 必须分层。
8. 学习记录、内容包、账号同步、支付权益、后台管理、监控、埋点、测试都必须纳入最终交付。

执行方式：
- 先阅读两份文档。
- 先输出阶段计划和当前阶段范围。
- 每阶段只实现该阶段目标。
- 每阶段结束必须给出修改文件、测试命令、测试结果、风险和下一步。
- 如果发现文档缺少必要细节，先提出缺口并给出合理默认方案，但不得擅自缩减商业级范围。

第一阶段从工程骨架、设计系统、数据库和内容包导入开始。
```

### 13.5 阶段通过门槛

每个阶段必须满足：

| 门槛 | 要求 |
| --- | --- |
| 编译 | Debug 构建通过 |
| 运行 | 真机或模拟器可进入本阶段页面 |
| 测试 | 至少有本阶段核心单元测试或 UI 测试 |
| 数据 | 不使用散落在 UI 的假数据 |
| 错误 | 关键错误有可见提示或日志 |
| 文档 | 更新本阶段完成状态和已知问题 |
| 回归 | 不破坏上一阶段核心流程 |

没有达到门槛，不进入下一阶段。

### 13.6 Codex 多子任务执行方案

计划使用 Codex 多子任务并行推进时，建议按职责拆分，而不是让多个子任务同时修改同一批文件。

推荐子任务：

| 子任务 | 职责 | 主要目录 |
| --- | --- | --- |
| 架构总控 | 模块边界、接口约束、合并审查、质量门槛 | 全局 |
| Android 客户端 | Compose UI、学习流程、音频、设置、统计 | `app/`, `feature-*`, `core-design` |
| 本地数据与算法 | Room、内容包导入、FSRS、学习计划、同步队列 | `core-domain`, `core-data`, `core-database` |
| 后端与后台 | PocketBase、手机号登录、同步、支付、后台集合 | `backend/` |
| 内容管线 | 中考词库、内容包格式、来源登记、审核表 | `content/`, `tools/content-*` |
| 测试与质量 | 单元测试、UI 测试、真机清单、验收报告 | `test/`, `docs/qa` |
| 合规与发布 | 隐私协议、许可证、上架材料、权限说明 | `docs/legal`, `docs/release` |

并行规则：

- 每个子任务开始前必须声明修改范围。
- 不同子任务不得同时修改同一模块核心文件。
- 架构总控负责合并前检查。
- 后端接口必须先出 OpenAPI 或接口文档，再由客户端接入。
- 内容管线不得把未授权词典内容写入正式内容包。
- 测试子任务不只写测试，也要运行测试并记录失败项。

推荐批次：

```text
批次 1：架构总控 + Android 骨架 + 数据库骨架 + 后端骨架
批次 2：内容包导入 + 首页/学习页 + PocketBase 集合
批次 3：FSRS + 学习流程 + 词典/收藏/笔记
批次 4：手机号登录 + 云同步 + 设置/统计
批次 5：支付/会员 + 后台管理 + 内容纠错
批次 6：埋点/监控 + 自动化测试 + 真机修复 + 上架材料
```

每个批次结束必须产出：

- 可运行版本。
- 变更摘要。
- 测试结果。
- 剩余问题。
- 下一批次任务拆分。

## 14. 8.5 分差距清单

如果 AI 做完后达不到 8.5 分，通常差在：

| 差距 | 表现 |
| --- | --- |
| UI 粗糙 | 页面像 Demo，状态少，间距不稳 |
| 内容弱 | 样例词少，释义和例句不可信 |
| 测试弱 | 只手动试过，没有自动化回归 |
| 同步弱 | 登录后数据合并和冲突处理不可靠 |
| 支付弱 | 只模拟会员，没有真实订单状态 |
| 后台弱 | 没有内容、反馈、配置、订单管理 |
| 监控弱 | 崩溃和关键错误不可见 |
| 合规弱 | 没有许可证、隐私、内容来源文件 |
| 音频弱 | 发音缺失、切换不稳定、失败无兜底 |
| 运营弱 | 没有埋点、灰度、内容包更新 |

补齐这些，才可能接近 8.5-9 分。

## 16. 结论

要达到 8.5-9 分，不能只靠一份产品文档让 AI “自由发挥”。必须把完整商业 App 拆成工程、内容、后台、商业、测试、合规六条线并行验收。

当前最合理策略：

1. 用《背单词App开发文档.md》作为产品总纲。
2. 用本规格书作为 AI 执行和验收标准。
3. 让 AI 分阶段实现，不允许一次性大包生成。
4. 每阶段都真机运行、测试、截图、修复。
5. 内容和版权人工介入。
6. 最终按 100 分评分表验收，低于 85 分不进入商业发布。
