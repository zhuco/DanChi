# DanChi 背单词 App 开发文档

版本：v0.2  
日期：2026-06-27  
阶段：商业级产品方案 + 技术方案 + Android MVP 开发基线  
首期核心词库：中考必备  
首期平台：Android 手机  
长期目标：先验证英语背单词商业价值，再扩展多词库、同步、iOS、小程序或其他语种

---

## 1. 项目定位

DanChi 是一款以 Android 为首发平台、以英语词汇学习为第一商业方向的背单词 App。首期不做泛英语学习平台，也不提前承诺日语、韩语、小程序或 iOS，而是先做一个聚焦中考词汇、能长期打磨并具备商业化基础的英语背单词产品。

首期定位：

- 面向初中生、中考备考用户、陪学家长。
- 以“中考必备词库”为第一词库。
- 以“科学复习 + 清晰词义 + 英美发音 + 例句语境 + 拼写巩固”为核心闭环。
- 先离线可用，再做账号同步、内容更新和商业化。
- 以商业级标准做 MVP：数据不能丢、内容可追溯、学习路径稳定、体验可持续迭代。

长期定位：

- 覆盖小学、初中、高中、大学四六级、专升本、考研、留学等词库。
- 建立自有词库内容资产，不依赖竞品或未授权词典内容。
- 支持自建词库、学习笔记、生词收藏、听写、跟读、释义选择、拼写测试。
- 形成“词库内容 + 复习算法 + 训练模式 + 用户学习数据”的商业壁垒。
- iOS、小程序、日语、韩语等方向仅作为英语产品验证后的独立战略决策，不在第一阶段牺牲 Android 英语 App 的交付效率。

## 2. 核心结论

1. 技术路线建议采用 Android 原生：Kotlin + Jetpack Compose + Room + FSRS。
2. 开源项目可以参考和局部二开，但不应直接把通用 flashcard 项目当作最终业务模型。
3. 首期不要复制有道、牛津、金山、柯林斯、剑桥、韦氏等词典内容。除非拿到正式授权，否则只能做“第三方词典跳转/查询入口”，不能内置其释义、例句、音频或品牌内容。
4. 首期内容库建议使用可商用许可证清晰的开源词典数据、官方课程标准、公开考试词频、自研例句和 TTS/授权音频。
5. 竞品截图显示的产品结构值得借鉴：今日计划为主路径，自由练习为辅助路径，词典/统计/我的为底部导航。
6. 中考 MVP 最重要的不是功能堆满，而是把每天学习的阻力降到最低：打开即学、发音稳定、释义简洁、复习量可控、错词可追踪。
7. 商业级 App 不是“功能多”就够了，必须同时具备内容生产流程、版本更新机制、数据埋点、用户反馈、崩溃监控、隐私合规和可灰度发布能力。
8. 未来如果做 iOS 或其他语种，可以重新评估 SwiftUI、Kotlin Multiplatform、Flutter 或独立原生方案；当前只要求核心业务代码不要和 Android UI 强耦合。
9. 如果目标不是 MVP，而是 8.5-9 分商业候选版本，应同时使用《DanChi商业完整版交付规格书.md》作为 AI 实施和验收标准。

### 2.1 当前明确不做

第一阶段不做以下内容，避免产品失焦：

- 不做日语、韩语等其他语种。
- 不做 iOS 客户端。
- 不做微信小程序。
- 不做复杂社交、排行榜、班级 PK。
- 不做大规模图片记忆库。
- 不做助记视频内容。
- 不做强登录，不把学习闭环锁在账号系统之后。

### 2.2 当前必须预留

虽然第一阶段只做 Android 英语 App，但必须预留以下能力：

- 词库内容和 App 代码分离，便于后续在线更新。
- 学习记录采用稳定数据结构，便于后续同步。
- 复习算法、学习计划、判分逻辑放在 domain 层，不写进 Compose 页面。
- 音频、例句、释义、助记内容都记录来源和版本。
- UI 文案、学习模式、词库配置集中管理，避免后续扩展时全局搜索硬改。

### 2.3 未来多语种原则

日语、韩语和英语的学习路径差异很大，未来即使做多语种，也不默认共用同一套 UI。

原则：

- 英语产品先独立验证商业价值。
- 日语、韩语如要做，优先作为独立 App 或独立产品线评估。
- 可复用账号、支付、内容后台、埋点、客服、部分学习算法思想。
- 不强行复用英语单词页 UI。
- 不为未来多语种过早抽象当前英语数据模型和页面。
- 当英语 App 有稳定收入后，再根据团队、内容成本和获客成本决定技术方向。

## 3. 本地截图竞品分析

截图位置：`E:\DanChi\doc`

共 8 张截图，疑似来自“疯狂背单词”类产品。以下分析只用于产品理解，不用于复制 UI 细节、品牌、图标、文案或数据内容。

### 3.1 首页

截图：`微信图片_20260627231805_55_146.jpg`

观察：

- 顶部品牌标题“疯狂背单词”。
- 首页第一块是“我的词库”，显示掌握、已学、总词量。
- 有“查看计划”入口。
- 网络不可用时在首页直接提示。
- “今日计划”模块显示待新学、待复习。
- 主按钮为“新学”和“复习”。
- “自由练习”以听、说、读、写四个方向展示：
  - 听力训练
  - 跟读对比
  - 释义巩固
  - 拼写练习
- 底部还有列表刷词、助记视频、随身听、学习记录、单词听写。
- 底部导航为：单词、统计、词典、我的。

可借鉴点：

- 首页应直接承载学习任务，不做营销落地页。
- 今日学习入口必须非常醒目。
- 练习模式用“听说读写”组织，比技术名词更容易理解。
- 网络状态、学习计划、词库进度要在首页显性化。

需要改进点：

- 首页功能入口偏多，初中用户可能分心。
- 当前实现应只保留一个主学习入口“开始提分”，由 FSRS 自动合并新词和到期复习；拼写、听写、释义选择、词典查询作为自由专项练习。
- “助记视频”等内容成本高，后期再做。

### 3.2 单词学习页

截图：`微信图片_20260627231800_51_146.jpg`

观察：

- 顶部显示当前进度 `0/10`。
- 顶部功能按钮包括熟词、收藏、笔记、设置。
- 单词主体使用大字号，例如 `a`。
- 发音区包含：
  - 美式/英式切换
  - 音标
  - 播放按钮
  - 发音指导
- 释义区显示词性和中文释义。
- 例句区展示英文例句、中文翻译、例句音频。
- 扩展内容区展示联想记忆。
- 词典区展示多个外部词典入口。
- 底部主按钮是“下一词”。

可借鉴点：

- 单词页层级清楚：单词 -> 发音 -> 释义 -> 例句 -> 助记 -> 词典。
- 收藏、笔记、熟词应作为高频操作放在学习页。
- 例句音频是重要体验点，不应只播放单词。
- 英美发音切换必须前置，不能藏在深层设置里。

需要改进点：

- 外部词典品牌入口必须解决授权问题。没有授权时不要展示品牌 Logo。
- 对中考用户，例句要控制难度，避免例句里出现大量超纲词。
- 助记内容要做质量审核，避免低俗谐音、错误词根、误导性联想。

### 3.3 退出确认弹窗

截图：`微信图片_20260627231801_52_146.jpg`

观察：

- 用户退出学习流程时弹出提示：“今天就学到这里了吗？”
- 主按钮是“继续学习”。
- 次按钮是“确认结束”。
- 弹窗文案强调按计划学习更利于坚持。

可借鉴点：

- 学习中退出是高流失节点，应温和挽留。
- 主行动应放在继续学习，不应过度惩罚用户。

建议：

- DanChi 可在退出时显示“还剩 3 个新词，预计 1 分钟”，让用户基于成本判断。
- 不要制造焦虑，不做过强的负面提醒。

### 3.4 熟词确认弹窗

截图：`微信图片_20260627231802_53_146.jpg`

观察：

- 用户点击“熟”后弹出二次确认。
- 文案说明“标记掌握后，将永远不再安排复习”。
- 提供“今天不再提示”选项。

可借鉴点：

- “熟词/掌握”是高风险操作，应二次确认。
- 用户需要控制提醒频率。

建议：

- DanChi 不建议默认“永远不复习”。更合理的是：
  - 标记熟词后移出日常复习。
  - 仍可在“熟词本”里手动恢复。
  - 每 90/180 天可选做一次抽查。

- DanChi 首期模式：
  - 轻松：每日 8-10 新词，复习上限为新词 3-5 倍。
  - 标准：每日 15 新词，复习上限为新词 5-7 倍。
  - 冲刺：每日 25 新词，复习上限动态上调。
- 复习量不能机械固定为新词倍数，应结合到期词和用户压力上限。

## 4. 竞品和市场模式

### 4.1 疯狂背单词类产品

公开 App Store 信息显示，该类产品强调基于遗忘曲线进行记忆规划，在单词濒临遗忘前安排复习，并支持系统词库和自定义词库。

对应策略：

- DanChi 不要只说“艾宾浩斯”，应落地为可解释的今日计划和复习压力控制。
- 中考用户不需要复杂参数，需要知道“今天学什么、为什么复习这些、还剩多久完成”。

### 4.2 墨墨背单词

墨墨主打抗遗忘、记忆模型、图表统计、词书兼容、助记库和云词本。App Store 描述中提到遗忘曲线、学习情况、记忆持久度等统计系统。

对应策略：

- DanChi 的第一差异点可以是“更轻的中考复习规划”。
- 首页应展示待学、待复习、预计完成时间、连续学习。
- 统计首期不必复杂，但必须能看见进度和薄弱词。

### 4.3 不背单词

不背单词主打真实语境、原声例句、词组搭配、词根词缀、正版柯林斯、全阶段词库、真题例句、拼写测试和打卡。

对应策略：

- DanChi 中考版应重视例句难度和考试义。
- 首期例句不追求数量，而追求“中考可理解、可听、可复用”。
- 拼写测试对中考作文有价值，应进入 MVP。

### 4.4 百词斩

百词斩长期强调图片记忆和轻量化学习，用户评价里也常提到图文混记的优势。

对应策略：

- DanChi 首期不建议做大规模图片库，成本高且版权风险大。
- 可以后期做少量“高混淆词图片/场景图”，但不作为 MVP 核心。

## 5. 用户痛点与解决方案

说明：本节结合本地截图、竞品 App Store 公开描述、公开用户评论关注点和背单词产品常见使用场景整理，不等同于大样本统计调研。正式商用前建议再做 20-50 个目标用户访谈，尤其要覆盖初中生、家长和英语老师。

### 5.1 复习压力不可控

痛点：

- 用户一开始热情高，后面到期复习量暴涨。
- 复习列表越积越多，产生挫败感。
- 学生不知道为什么今天要复习这些词。

解决方案：

- 使用 FSRS 或同类间隔重复算法，为每个词维护独立记忆状态。
- 加入复习压力上限，不让每日任务失控。
- 首页显示“预计完成时间”和“今日负荷”。
- 长期未学时提供“恢复计划”，而不是一次性堆积所有逾期词。

### 5.2 只会认，不会听写和拼写

痛点：

- 很多背词 App 让用户以为自己认识，但考试作文和听力里用不出来。
- 看词选义过于轻松，实际掌握度不足。

解决方案：

- 每个词至少经历三类训练：
  - 看词回忆释义
  - 看义/听音拼写
  - 听音辨义
- 中考高频词进入拼写强化队列。
- 用户多次拼写错误的词自动加入“易错拼写本”。

### 5.3 例句太难或脱离考试

痛点：

- 例句里出现很多超纲词，初中生读不懂。
- 例句过于成人化或书面化，记不住也用不上。

解决方案：

- 中考必备词库的例句按 CEFR A1-B1、初中语境、短句优先。
- 例句字段要标注难度、来源、是否含超纲词。
- 首期每词 1-2 条核心例句，宁少勿滥。
- 例句音频和中文翻译同步提供。

### 5.4 词典内容权威但版权风险高

痛点：

- 用户喜欢牛津、剑桥、柯林斯、韦氏等权威词典。
- 但这些词典数据一般需要授权，不能随意抓取或内置。

解决方案：

- 首期内置自有简明释义和例句。
- 可提供“第三方词典查询”外部跳转，不内置其内容和 Logo。
- 若后期付费商业化，再洽谈 Oxford、Cambridge、Collins、Merriam-Webster 等 API 或数据授权。
- 所有词库内容建立来源登记表，记录许可证、授权范围、更新时间。

### 5.5 发音体验不稳定

痛点：

- 单词发音和例句发音经常缺失。
- 英式/美式切换不明显。
- 音频播放太快，学生听不清。

解决方案：

- 发音字段必须区分 `us` 和 `uk`。
- 支持默认发音偏好：美式 / 英式。
- 支持播放速度：0.75x / 1.0x / 1.25x。
- 首期使用系统 TTS 或云 TTS 生成音频缓存，后期替换为授权真人音频。
- 音频播放失败时回退 TTS。

### 5.6 助记内容质量参差

痛点：

- 谐音记忆容易低俗、牵强、误导发音。
- 词根词缀内容不严谨，容易造成错误理解。

解决方案：

- 助记内容分类型：
  - 词根词缀
  - 发音联想
  - 场景联想
  - 近反义/词族
- 首期只内置审核过的少量内容。
- 用户自定义助记只默认自己可见，社区化后再做审核。

### 5.7 付费和广告破坏学习专注

痛点：

- 学习 App 过度弹会员、广告或活动，会伤害学生专注。
- K12 用户对诱导付费更敏感。

解决方案：

- MVP 不做广告。
- 未来商业化以高级词库、云同步、AI 讲解、真人音频、家长报告为权益。
- 不把基础学习闭环锁死在付费墙后。

### 5.8 自建词库和笔记不方便

痛点：

- 学生老师常有自己的单词清单、错题单词、课堂补充词。
- 很多 App 自建词库入口深、导入格式复杂。

解决方案：

- 首期支持手动添加单词和笔记。
- 第二阶段支持 CSV 导入。
- 第三阶段支持文本粘贴提词、拍照 OCR、文章提词。
- 自建词库也走同一套复习算法。

### 5.9 商业级切入机会

首期选择“中考必备”不是因为市场最大，而是因为范围可控、需求刚性、用户场景清晰。初中英语词汇学习有明确考试周期，家长愿意为学习效率和确定性付费，内容体量也适合小团队先做深。

机会点：

- 成熟 App 往往覆盖面很广，但对中考用户未必足够轻。
- 初中生需要更短例句、更少参数、更明确的每日计划。
- 家长关心“有没有坚持、有没有进步、哪些词薄弱”，不一定需要复杂社交。
- 老师和学生经常有课堂补充词、错题词，自建词库和笔记有真实需求。
- 词库版权和内容可信度是商业化门槛，也是可形成壁垒的地方。

首期差异化建议：

- “中考词义优先”：不追求词典大而全，优先展示考试常用义。
- “复习压力可控”：不让逾期词把用户压垮。
- “拼写和听写真实掌握”：避免只会看词选义。
- “例句适合初中”：例句短、可理解、可朗读、可用于作文。
- “少打扰”：不在学习流程中穿插广告、活动弹窗或强付费提示。

是否继续投入的判断指标：

| 指标 | 目标方向 |
| --- | --- |
| 新用户首日完成率 | 用户安装后能完成一次学习 |
| 7 日留存 | 用户愿意持续回来复习 |
| 今日任务完成率 | 计划难度和流程阻力是否合理 |
| 复习逾期率 | 复习压力是否失控 |
| 拼写正确率变化 | 是否真正提升掌握度 |
| 内容纠错率 | 词库质量是否稳定 |
| 付费意愿访谈 | 家长/学生是否认可高级能力 |

## 6. 合规与内容版权原则

### 6.1 不能直接使用的内容

未经授权，不应内置或复制以下内容：

- 有道词典释义、例句、音频、品牌 Logo。
- 牛津词典释义、例句、音频、品牌 Logo。
- 金山词霸/爱词霸释义、例句、音频、品牌 Logo。
- 柯林斯词典释义、例句、音频、品牌 Logo。
- 剑桥词典释义、例句、音频、品牌 Logo。
- 韦氏词典释义、例句、音频、品牌 Logo。
- 其他背单词 App 的词库、例句、助记、图片、音频。
- 商业教材、教辅书、词书中的独家释义、编排和例句。

### 6.2 可以采用的内容策略

首期推荐：

- 单词清单：基于《义务教育英语课程标准（2022年版）》、公开考试要求、公开中考高频词、ECDICT 中考标签和人工审核整合。
- 简明释义：使用许可证清晰的开源词典作为初稿，再人工修订为中考学习释义。
- 英文释义：首期可以不展示，避免难度过高。
- 例句：自研或 AI 生成后人工审核，不直接抓取竞品或商业词典。
- 音频：系统 TTS / 云 TTS / 后期授权音频。
- 词根词缀：自研小型内容库，优先覆盖中考高频可解释词。

### 6.2.1 词库数据授权边界

不是所有单词数据都需要商业授权。需要区分“单词事实/自有词表”和“第三方词典内容”。

| 数据类型 | 授权判断 | 处理建议 |
| --- | --- | --- |
| 英文单词本身 | 通常不需要商业授权 | 可以使用 |
| 自己整理的中考词表 | 不一定需要 | 基于公开标准和自有筛选生成，不复制竞品顺序和分类 |
| 公开课程标准和考试范围 | 看来源和使用方式 | 可参考归纳，避免整篇复制受保护材料 |
| 开放许可词典数据 | 按许可证 | 记录许可证、署名、商用条件 |
| 第三方商业词典释义 | 通常需要授权/API 许可 | 未授权不得批量内置 |
| 第三方商业词典例句/搭配/短语 | 通常需要授权/API 许可 | 未授权不得批量内置 |
| 第三方商业词典音频 | 需要授权 | 未授权不得下载内置 |
| 第三方品牌 Logo 和“内置某词典”宣传 | 需要授权或品牌许可 | 未授权不要展示 |
| 自研释义、例句、助记、TTS 音频 | 不需要第三方授权 | 需要内部审核和来源记录 |

因此，DanChi 可以自建中考必备词表和自有内容。版权风险主要来自复制商业词典的释义、例句、音频、编排和品牌资产，而不是来自英语单词本身。

### 6.3 内容来源登记表

每条内容资产应记录：

| 字段 | 说明 |
| --- | --- |
| content_id | 内容唯一 ID |
| content_type | word / definition / example / audio / mnemonic / root |
| source_name | 来源名称 |
| source_url | 来源链接 |
| license | 许可证 |
| commercial_allowed | 是否允许商用 |
| attribution_required | 是否需要署名 |
| modified | 是否修改 |
| reviewer | 审核人 |
| reviewed_at | 审核时间 |

### 6.4 商业级内容生产流程

中考词库不能只做一次性整理，应按可持续运营流程建设。

内容流程：

```text
候选词收集
  -> 来源登记
  -> AI 生成释义/例句/搭配/短语/同反义词初稿
  -> 自动校验和相似度检查
  -> TTS 音频生成/绑定
  -> 一审：拼写、词性、释义
  -> 二审：中考适配、例句难度、版权风险
  -> 内容版本打包
  -> 灰度发布
  -> 用户纠错回流
  -> 下个版本修订
```

每次内容发布必须生成：

- 内容包版本号，例如 `zk_core_2026.07.01`。
- 词条新增、修改、删除清单。
- 释义和例句变更清单。
- 音频缺失清单。
- 来源和许可证变更清单。
- 回滚包。

### 6.4.1 AI 生成内容策略

可以使用 AI 生成官方词库内容初稿，包括：

- 中文核心释义。
- 中考常用义排序。
- 英文例句和中文翻译。
- 搭配。
- 短语。
- 同义词。
- 反义词。
- 易混词。
- 词根词缀解释。
- 谐音/联想助记。
- 考点提示。
- 拼写易错点。

原则：

- AI 生成内容必须进入审核流程，不能直接发布。
- 提示词中不得要求复刻有道、牛津、金山、柯林斯、剑桥、韦氏等词典原文。
- 不要求 AI 模仿某个词典风格。
- 例句要适合初中语境，避免明显超纲、成人化或敏感内容。
- 同反义词、搭配、短语要做交叉验证，低置信度内容不进入正式包。
- 谐音助记不得低俗，不得误导发音。

自动校验：

- 拼写校验。
- 词性校验。
- 例句目标词校验。
- 例句长度和超纲词校验。
- 敏感内容校验。
- 与第三方词典内容的相似度检查。
- 同反义词和搭配合理性检查。
- 中文翻译漏译/错译检查。

审核分级：

| 内容 | 审核强度 |
| --- | --- |
| 核心释义 | 必须人工审核 |
| 例句和翻译 | 必须人工审核 |
| 中考核心词搭配/短语 | 必须人工审核 |
| 同反义词 | 抽检 + 高频词必审 |
| 词根词缀 | 必须人工审核 |
| 谐音/联想助记 | 必须人工审核 |

### 6.4.2 TTS 音频策略

音频可以直接使用 TTS，不需要使用第三方词典真人音频。

建议：

- MVP/内测：Android 系统 TTS 作为兜底。
- 商业候选版：使用可商用 TTS 服务预生成高频词英美音频和例句音频。
- 音频 manifest 记录 voice、accent、provider、license、生成时间、checksum。
- 美式/英式发音分开生成或分开调用。
- TTS 生成失败时，App 端回退系统 TTS。

注意：

- 必须确认 TTS 服务允许商业使用和 App 内分发。
- 不使用克隆真人声音。
- 不使用“真人词典音频”等误导性文案。
- 如果未来提供用户实时 AI 生成例句/讲解，应按生成式 AI 内容安全和标识要求单独设计。

### 6.5 内容质量标准

中考必备词库 v1 的质量标准：

- 核心词拼写错误率为 0。
- 词性和中文释义必须经过人工审核。
- 例句尽量短句，避免明显超纲词堆叠。
- 例句不出现暴力、低俗、歧视、政治敏感、成人化场景。
- 谐音助记不误导发音，不使用低俗联想。
- 每个词至少有一个可离线展示的核心释义。
- 音频缺失时必须有 TTS 兜底。

### 6.6 用户纠错机制

商业级版本必须提供“单词纠错”入口，至少支持：

- 释义错误。
- 例句错误。
- 翻译不自然。
- 发音问题。
- 词性错误。
- 其他问题。

纠错数据进入后台后应记录：

- 用户 ID 或本地匿名 ID。
- 词条 ID。
- 问题类型。
- 用户描述。
- App 版本。
- 内容包版本。
- 提交时间。
- 处理状态。

## 9. 核心用户流程

### 9.1 首次启动

1. 用户打开 App。
2. 选择首期词库：默认“中考必备”。
3. 设置每日新词量：默认 10。
4. 设置默认发音：美式 / 英式。
5. 进入首页，展示今日计划。

首次启动不要强制登录。

### 9.2 每日学习

1. 用户进入首页。
2. 看到待新学、待复习、预计完成时间。
3. 点击“开始学习”。
4. 系统按规则混合新词和复习词。
5. 每个词进入学习卡片。
6. 用户选择掌握程度或进入下一题型。
7. 完成后进入学习完成页。

### 9.3 单词学习卡片

默认展示：

- 单词
- 发音按钮
- 音标

用户点击后展示：

- 中文释义
- 例句
- 例句音频
- 助记
- 收藏/笔记/熟词

评分按钮：

- 忘记
- 模糊
- 认识
- 熟练

底层映射：

- 忘记 -> Again
- 模糊 -> Hard
- 认识 -> Good
- 熟练 -> Easy

### 9.4 拼写测试

触发条件：

- 新学完成后抽测。
- 复习达到一定熟悉度后进入拼写。
- 写作高频词必须进入拼写强化。

题型：

- 中文释义 -> 输入英文。
- 听音 -> 输入英文。
- 例句填空 -> 输入目标词。

### 9.5 释义选择

触发条件：

- 新词初学。
- 复习时快速确认。

题型：

- 看英文选中文。
- 看中文选英文。
- 听音选中文。

### 9.6 收藏和生词本

收藏：

- 用户主动收藏。
- 方便反复查看。

生词本：

- 用户查词后加入。
- 学习流程中标记“忘记”多次自动加入。
- 可单独复习。

### 9.7 熟词处理

用户点击熟词：

1. 弹出确认。
2. 说明：熟词将从每日任务中移除，但可在熟词本恢复。
3. 用户确认后，状态改为 `suspended`。

不要做不可恢复的删除。

- 更多词库：小学、高中、四级、六级、考研、专升本

### 10.4 商业级埋点需求

MVP 阶段就应加入基础埋点，但不要采集无关隐私数据。埋点目标是判断学习闭环是否成立、内容是否有效、复习压力是否合理。

核心事件：

| 事件 | 触发时机 | 关键属性 |
| --- | --- | --- |
| app_open | 打开 App | app_version, content_version |
| onboarding_complete | 完成首次设置 | daily_new_words, default_accent |
| study_start | 开始学习 | wordbook_id, new_count, review_count |
| word_card_show | 展示单词卡 | word_id, mode, queue_type |
| pronunciation_play | 播放发音 | word_id, accent, source, success |
| review_submit | 提交记忆反馈 | word_id, rating, response_ms |
| spelling_submit | 提交拼写 | word_id, correct, attempt_count |
| choice_submit | 提交选择题 | word_id, correct |
| study_finish | 完成学习 | learned_count, reviewed_count, duration |
| word_favorite | 收藏单词 | word_id, source_page |
| note_save | 保存笔记 | word_id |
| dictionary_search | 查词 | query, hit |
| content_report | 内容纠错 | word_id, issue_type |
| plan_change | 修改计划 | daily_new_words, review_limit |

关键指标：

- 激活：首次学习完成率。
- 留存：D1、D7、D30 留存。
- 学习：每日任务完成率、平均学习时长、平均新词量。
- 复习：逾期词比例、复习完成率、Again 比例。
- 内容：查词命中率、纠错率、音频播放失败率。
- 训练：拼写正确率、选择题正确率、听写完成率。
- 稳定：崩溃率、ANR、数据库迁移失败率。

### 10.5 功能开关和灰度能力

商业级 App 需要预留功能开关，避免所有能力一次性对全量用户开放。

首期可做本地配置，后期接远程配置：

- 是否开启拼写测试。
- 是否开启听写训练。
- 是否展示助记内容。
- 是否展示第三方词典跳转。
- 是否开启新学习流程实验。
- 是否开启内容包更新。
- 是否展示付费入口。

灰度发布要求：

- 新内容包先对内部测试用户开放。
- 关键流程改动先对少量用户开放。
- 出现崩溃或内容错误时可以远程关闭入口。

## 11. 非功能需求

### 11.1 性能

- 首页冷启动 2 秒内可见主要内容。
- 单词切换无明显卡顿。
- 查询本地词典 200ms 内返回。
- 学习流程必须离线可用。
- 音频缓存命中时 300ms 内开始播放。

### 11.2 稳定性

- 学习记录必须本地持久化。
- App 崩溃后不丢失当前学习进度。
- 数据库升级必须有迁移测试。
- 音频失败不阻塞学习流程。

### 11.3 可用性

- 中学生可在 30 秒内理解首页。
- 今日学习路径最多 2 次点击进入。
- 默认设置可直接使用，不要求用户理解复杂算法。
- 所有核心按钮支持单手操作。

### 11.4 隐私

- MVP 不强制注册。
- 学习数据默认保存在本地。
- 后续云同步前明确告知数据用途。
- K12 商业化时避免不必要的社交和排行榜。

### 11.5 商业级稳定性指标

上线前建议达到：

- Crash-free sessions 不低于 99.5%。
- 学习记录写入失败率接近 0。
- 词库导入失败必须可重试。
- 内容包更新失败不能影响已下载词库学习。
- 音频播放失败不能阻塞下一步学习。
- 数据库迁移必须有自动化测试和回滚预案。

### 11.6 可观测性

MVP 后期应接入：

- 崩溃日志。
- ANR 监控。
- 音频播放失败日志。
- 内容包下载失败日志。
- 数据库迁移失败日志。
- 关键学习流程漏斗。

日志原则：

- 不记录用户输入的完整笔记内容。
- 不记录未脱敏手机号、邮箱等敏感信息。
- 不上传完整本地词库内容。
- 错误日志必须包含 App 版本、数据库版本、内容包版本。

### 11.7 数据备份与恢复

学习数据是用户核心资产，必须可恢复。

MVP：

- 本地数据库持久化。
- 手动导出学习数据。
- 数据库升级前自动备份关键表。

商用版：

- 账号云同步。
- 多设备冲突合并。
- 用户主动注销和数据删除。
- 内容包和学习记录分开备份。

### 11.8 未成年人和家长场景

面向中考用户时，产品设计要更克制：

- 不做诱导性消费弹窗。
- 不做现金奖励、赌博式抽奖、过强连续签到惩罚。
- 不公开展示未成年人的学习排名。
- 后期家长报告只展示学习进度和薄弱项，不做羞辱式文案。
- 隐私协议和用户协议应为正式上线前必要项。

### 12.2 分层结构

```text
android-app/
  app/                  启动、导航、依赖注入、全局配置
  feature-home/         首页和今日计划
  feature-study/        新学、复习、拼写、释义选择
  feature-dictionary/   查词、词典详情
  feature-statistics/   学习统计
  feature-profile/      我的、设置、反馈
  core-design/          主题、组件、图标、排版
  core-domain/          纯业务模型、FSRS、学习计划、判分
  core-data/            Repository、Room、内容包导入
  core-audio/           音频播放、TTS、缓存
  core-analytics/       埋点、崩溃和日志封装
  core-common/          时间、结果类型、错误处理
```

分层原则：

- Compose 页面只处理 UI 状态和用户交互，不直接操作数据库。
- FSRS、学习计划、答题判分放在 `core-domain`。
- Room Entity 不直接暴露给 UI。
- 内容包导入逻辑独立，不散落在页面里。
- 埋点封装为接口，避免业务代码绑定某个第三方 SDK。
- 音频播放封装为服务，学习页只调用播放意图。

### 12.3 数据流

```text
Compose UI
  -> ViewModel
  -> UseCase
  -> Repository
  -> Room DAO / Audio Service / Scheduler
  -> StateFlow
  -> Compose UI
```

### 12.4 为什么选择 Android 原生

- 首期目标是 Android 手机，原生体验优先。
- 发音、通知、离线数据库、系统 TTS、后台任务在 Android 原生生态更稳。
- Jetpack Compose 适合快速构建现代 Android UI。
- 当前阶段不为 iOS 或多语种提前牺牲交付效率。
- 未来是否迁移或复用核心代码，应由英语 App 的商业数据决定。

### 12.5 未来平台边界

后期可能出现 iOS、小程序、日语、韩语等方向，但当前不做技术承诺。架构只需做到“边界清楚”，而不是提前做过度抽象。

当前必须做到：

- 业务核心不写进 UI 页面。
- 内容包格式不绑定 Android。
- 学习记录结构稳定。
- API DTO 和本地 Entity 分离。
- UI 组件服务英语首期体验，不强行做多语种通用 UI。

未来可选方向：

| 方向 | 触发条件 | 技术再评估 |
| --- | --- | --- |
| iOS | Android 证明留存和付费价值 | SwiftUI / Compose Multiplatform / Flutter |
| 微信小程序 | 需要轻量查词、打卡、家长报告 | Taro / 原生小程序 / uni-app |
| 日语 App | 英语产品形成稳定收入和内容流程 | 独立 UI + 复用部分后台 |
| 韩语 App | 日语或英语验证多语种商业价值 | 独立 UI + 复用部分后台 |

不建议现在做：

- 为未来多语种把当前英语单词页做成过度通用模板。
- 为 iOS 过早引入复杂跨平台 UI。
- 为小程序牺牲 Android 离线、音频和通知体验。

### 12.6 后端演进

MVP 可以无后端或弱后端，但商用前需要后端能力。

当前后端选型：

| 层 | 选型 |
| --- | --- |
| 后端核心 | PocketBase + Go 扩展 |
| 后台管理 | PocketBase Admin Dashboard 为主 |
| 数据库 | SQLite 起步，定期备份 |
| 账号 | 手机号 + 短信验证码 |
| 支付 | 微信支付 + 支付宝 App 支付 |
| 短信 | 阿里云短信或腾讯云短信，封装供应商接口 |

选择原因：

- PocketBase 单文件部署，内置 SQLite、Admin Dashboard、用户、文件和 REST API，服务器资源占用低。
- 首版后台不需要从零开发完整 CMS，可先用 PocketBase collections 管理内容、反馈、订单和配置。
- 支付、短信、同步、内容包发布通过 Go 自定义路由和 hooks 补齐。
- 如果后期用户规模或内容审核流程变复杂，再评估 PostgreSQL、Directus 或独立微服务。

阶段设计：

```text
MVP
  本地词库 + 本地学习记录 + 手动导出

内测
  内容包更新服务 + 反馈提交 + 崩溃监控

公测
  账号系统 + 云同步 + 会员权益 + 远程配置

商用
  内容管理后台 + 支付订单 + 客服工单 + 数据看板
```

后端核心模块：

- 手机号账号。
- 学习记录同步。
- 内容包版本管理。
- 用户反馈和内容纠错。
- 会员权益。
- 微信/支付宝支付订单。
- 远程配置。
- 运营数据看板。

## 13. 复习算法设计

### 13.1 算法选择

当前实现使用开源 `java-fsrs`。Open Spaced Repetition 社区说明 FSRS 已有多语言实现，并以 Difficulty、Stability、Retrievability 等记忆状态建模。

首期不手写简化版 FSRS 公式：

- 一个 `word` 可以对应多张 `card`，调度对象是 `card`，不是单词本身。
- 第一版至少实现 `recognition` card：英文单词 -> 4 选 1 中文释义。
- 每张卡维护 `state`、`difficulty`、`stability`、`due_at`、`reps`、`lapses`。
- 用户答题后映射为 FSRS 的 `Again/Hard/Good/Easy` 四档 rating，再由开源 FSRS 生成下次到期时间。
- 先使用 FSRS 默认参数，后期根据用户数据通过外部优化器优化参数。

### 13.2 状态定义

| 状态 | 说明 |
| --- | --- |
| new | 未学习 |
| learning | 当天学习中 |
| review | 正常复习 |
| relearning | 遗忘后重新学习 |
| suspended | 可作为运营/用户暂停状态，不作为首版主调度状态 |

### 13.3 用户反馈映射

| 答题结果 | 算法评分 | 含义 |
| --- | --- | --- |
| 首答错误 | Again | 需要重学 |
| 首答正确但用提示或耗时过长 | Hard | 有印象但不稳 |
| 首答正确且普通速度 | Good | 正常掌握 |
| 复习阶段连续快答且无提示 | Easy | 很熟悉 |

### 13.4 复习压力控制

规则：

- 队列优先级：learning/relearning 到期卡片 > review 到期卡片 > 新词。
- 所有复习以 `card.due_at` 为准，不使用固定艾宾浩斯天数表。
- 如果到期卡片预计耗时超过每日学习时长，当日不补新词。
- 不把大量逾期词一次性压给用户。

### 13.5 掌握判定

一个词不应只因一次 Easy 就永久掌握。

建议条件：

- 最近 3 次复习至少 2 次 Good/Easy。
- 最近一次拼写正确。
- 当前稳定度超过阈值。
- 用户没有主动标记易错。

## 15. UI/UX 设计原则

### 15.1 视觉方向

建议：

- 白色和浅灰作为主体。
- 黄色作为主行动色，但不要直接复制竞品色值。
- 大字号单词，清晰层级。
- 卡片半径控制在 8dp 左右，避免过度可爱化。
- 面向中考用户，风格应干净、可信、少干扰。

### 15.2 首页设计

首页第一屏应包含：

- 当前词库：中考必备
- 总进度：已学 / 掌握 / 总词数
- 今日任务：新词 N，到期 M
- 预计完成时间：约 X 分钟
- 主按钮：开始提分
- 次入口：拼写、听写、选义、词典

### 15.3 单词页设计

优先级：

1. 单词本身
2. 发音和音标
3. 中文核心释义
4. 例句
5. 助记
6. 笔记/收藏
7. 更多词典

不要在一个页面堆太多入口，首屏要聚焦。

### 15.4 中考用户特别设计

- 例句短。
- 中文释义简洁。
- 拼写输入框足够大。
- 错词反馈具体，例如“漏了一个 m”。
- 听写可反复播放，不惩罚。
- 学习完成后给清楚的结果，而不是复杂图表。

## 17. 词典方案

### 17.1 MVP 内置词典

功能：

- 搜索单词。
- 展示中文释义。
- 展示音标。
- 展示发音。
- 展示例句。
- 加入收藏/生词本。
- 加入学习计划。

数据来源：

- ECDICT 或其他可商用许可清晰的数据作为基础。
- 中考核心词经过人工修订。
- 每条词义保留来源和审核记录。

### 17.2 第三方词典

未授权时：

- 只提供外部浏览器跳转。
- 不展示对方 Logo。
- 不缓存对方内容。
- 不宣称“内置牛津/柯林斯/剑桥/韦氏”。
- 不批量抓取有道、牛津、金山、柯林斯、剑桥、韦氏等词典的释义、例句、音频或图片。
- 不把未授权词典内容写入正式词库内容包，即使当前阶段暂不商用。

授权后：

- 通过官方 API 或数据包接入。
- 显示授权声明。
- 按协议处理缓存、调用量、署名和付费边界。

研发期允许：

- 使用官方 API 试用额度做技术验证。
- 外部跳转到第三方词典页面。
- 人工少量核对词义，不复制原文表达。
- 使用许可证明确的数据源和自研内容生成首版词库。

未来可评估授权/API：

- Oxford Dictionaries API。
- Cambridge Dictionary API。
- Collins Dictionary API。
- Merriam-Webster Dictionary API。
- 有道智云或有道词典开放能力。
- 金山词霸/爱词霸开放接口或商务授权。

## 19. 商业化设计

商业化要建立在学习闭环有效、内容可信、用户愿意持续使用的基础上。第一版不要急于收费，先验证中考用户是否真的能连续使用并完成复习。

### 19.1 免费层

免费应包含：

- 中考必备基础词库。
- 每日学习和复习。
- 基础发音。
- 收藏和笔记。
- 基础统计。

免费层原则：

- 免费用户必须能完成完整学习闭环。
- 不把每日复习、核心词库、基础发音锁死。
- 免费层要足够形成口碑，但高级效率工具可收费。

### 19.2 付费层

适合付费的内容：

- 更多官方整理词库。
- 高质量真人发音。
- AI 例句和 AI 讲解。
- 云同步。
- 家长报告。
- 深度统计。
- 高级训练模式。
- 无限制自建词库。

优先级建议：

| 权益 | 商业优先级 | 原因 |
| --- | --- | --- |
| 云同步 | 高 | 用户学习数据有资产属性 |
| 更多词库 | 高 | 内容包容易形成付费边界 |
| 高质量真人发音 | 中 | 有体验价值但成本较高 |
| 家长报告 | 中 | K12 付费决策者常是家长 |
| 深度统计 | 中 | 适合重度用户 |
| AI 例句/讲解 | 中低 | 成本和质量控制压力大 |
| 无限自建词库 | 中低 | 先验证真实使用率 |

### 19.3 不建议的商业化方式

- 强制看广告才能继续学习。
- 学到一半弹出付费。
- 基础复习功能收费。
- 诱导未成年人高频消费。

### 19.4 付费验证路径

不要一开始就做复杂会员体系。推荐按以下路径验证：

1. 内测阶段只做免费版本，验证留存和学习完成率。
2. 公测阶段加入“高级功能即将开放”入口，收集点击和问卷。
3. 访谈家长和学生，确认愿意付费的权益。
4. 小范围开放会员，先验证真实付费，不急于大规模推广。
5. 根据付费数据决定是否投入更多词库、真人音频、云同步。

关键验证问题：

- 家长愿意为什么付费：词库、报告、同步、音频还是 AI 讲解？
- 学生是否真的使用高级训练，还是只需要每日任务？
- 免费层是否已经足够留住用户？
- 付费入口是否打扰学习流程？

### 19.5 会员和内容包关系

建议先设计两类商品：

| 商品类型 | 说明 |
| --- | --- |
| 会员 | 云同步、深度统计、家长报告、高级训练 |
| 内容包 | 高考、四级、六级、考研、专项词库、真人音频包 |

注意：

- 首期中考必备基础词库建议免费。
- 后续扩展词库可以作为内容包或会员权益。
- 不建议把用户已经学过的词库突然改成付费不可用。
- 付费权益要能在离线状态下有合理处理。

支付实现：

- 首版支付方式采用微信支付和支付宝 App 支付。
- 支付签名、订单状态、回调验签必须在后端完成。
- App 端不能保存支付密钥、证书或私钥。
- 会员权益以后端订单和支付回调为准，不能只依赖客户端回调。
- 必须支持支付失败、取消、超时、退款、权益恢复。

账号实现：

- 首版账号方式采用手机号 + 短信验证码。
- 支持游客模式继续学习。
- 支持游客数据登录后合并到手机号账号。
- 短信发送必须做手机号频控和 IP 频控。

### 19.6 商业级运营能力

商用版本需要运营后台，不然内容、反馈、订单、用户问题都会不可控。

最低后台能力：

- 内容包上传和版本管理。
- 用户反馈和内容纠错处理。
- 会员订单查询。
- 用户学习数据概览。
- 远程配置和功能开关。
- 公告和版本升级提示。

运营指标：

- 新增用户。
- 激活率。
- D1/D7/D30 留存。
- 日任务完成率。
- 内容纠错数量。
- 付费转化率。
- 退款率。
- 客服问题类型分布。

## 20. 开源项目使用策略

### 20.1 可参考方向

可以参考 MIT/Apache 项目的：

- FSRS 调度流程。
- 卡片学习状态设计。
- 离线数据结构。
- 同步思路。
- 导入导出格式。

### 20.2 不建议直接照搬

不要直接照搬：

- GPL/AGPL 项目代码。
- 竞品 UI。
- 竞品词库。
- 竞品例句。
- 竞品音频。
- 竞品图标和品牌视觉。

### 20.3 如果二开 Flashcards Open Source App

评估步骤：

1. 确认 LICENSE 为 MIT。
2. 本地跑通 Android 客户端。
3. 梳理其数据模型是否适合词库业务。
4. 只保留可复用的底层能力。
5. 尽快改造为 DanChi 自有词库模型。
6. 保留原许可证声明。

判断标准：

- 如果改造成本低于自建 30%，可以二开。
- 如果数据模型和业务方向冲突，应自建 Android 工程，只借鉴算法和结构。

### 20.4 商业级开源审查清单

引入任何开源项目或依赖前，应记录：

| 项 | 要求 |
| --- | --- |
| 许可证 | MIT / Apache-2.0 优先，谨慎使用 GPL / AGPL |
| 最近维护 | 近 12 个月有提交或 release 更好 |
| 依赖数量 | 依赖链过深要谨慎 |
| 安全问题 | 检查已知漏洞和废弃依赖 |
| 商业限制 | 确认是否允许商用、修改、分发 |
| 署名要求 | 保留 NOTICE、LICENSE、版权声明 |
| 替换成本 | 核心依赖必须有替代方案 |

项目内应维护：

- `THIRD_PARTY_LICENSES.md`
- 开源依赖清单。
- 词库和内容来源清单。
- 修改过的开源代码说明。

## 21. 开发里程碑

### Phase 0：商业和技术验证，3-5 天

目标：

- 确认首期只做 Android 英语中考方向。
- 确认自建工程还是二开开源项目。
- 建立 Android Compose 空工程。
- 跑通 Room。
- 跑通单词卡片页面。
- 跑通本地词库导入。
- 跑通 TTS 发音。
- 接入开源 FSRS 调度。
- 确认 50 个样例词的数据格式和审核流程。

交付：

- 可安装 APK。
- 50 个测试词。
- 本地学习记录。
- 技术选型结论。
- 内容来源登记样表。

继续投入条件：

- 单词学习流程能在真机上稳定跑通。
- 数据结构能支持后续词库扩展。
- 二开项目没有明显许可证或架构阻碍。

### Phase 1：中考 MVP，2-4 周

目标：

- 中考必备词库 v1。
- 首页今日计划。
- 新学和复习流程。
- 单词详情。
- 英美发音切换。
- 例句展示。
- 拼写测试。
- 释义选择。
- 收藏和笔记。
- 基础统计。

交付：

- 内测 APK。
- 1,600-1,700 词内容库。
- 内容来源登记表。
- 数据库迁移脚本。
- 基础埋点事件。

继续投入条件：

- 内部连续使用 7 天不丢数据。
- 10 个新词学习流程在 6-8 分钟内完成。
- 词库内容错误可被发现和修正。
- 学习记录能正确影响下次复习。

### Phase 2：Alpha 内测，2 周

目标：

- 学习流程打磨。
- 错词本。
- 学习提醒。
- 音频缓存。
- 设置页。
- 退出挽留。
- 熟词确认。
- 导出备份。
- 内容纠错入口。
- 崩溃和关键错误日志。

交付：

- 20-50 人内测版本。
- 内测反馈表。
- 内容修订记录。
- 崩溃和学习漏斗数据。

继续投入条件：

- 内测用户能独立完成首次设置和每日学习。
- 没有高频数据丢失或崩溃问题。
- 用户反馈集中在内容和体验优化，而不是“不知道怎么用”。

### Phase 3：Beta 公测，4-6 周

目标：

- 账号系统。
- 云同步。
- 内容版本更新。
- 远程配置。
- 用户反馈后台。
- 基础运营数据看板。
- 隐私协议和用户协议。

交付：

- 公测 APK。
- 内容包更新服务。
- 反馈和纠错处理流程。
- 第一版数据看板。

继续投入条件：

- D1、D7 留存达到可继续优化的水平。
- 日任务完成率稳定。
- 内容纠错量可控。
- 用户对高级权益有明确兴趣。

### Phase 4：商用准备，4-8 周

目标：

- 会员体系。
- 支付接入。
- 更多词库。
- 客服流程。
- 订单和权益后台。
- 上架材料。
- 合规审查。

交付：

- 商用候选版本。
- 后台管理系统。
- 内容审核流程。
- 会员权益配置。
- 上架所需文案、截图、隐私协议。

继续投入条件：

- 免费学习闭环已经稳定。
- 付费权益经过小范围验证。
- 词库版权和内容来源可审计。
- 支付、退款、权益恢复流程可验证。

### Phase 5：扩展决策

只有当英语 Android 产品形成有效数据后，再评估：

- 是否做 iOS。
- 是否做微信小程序。
- 是否扩展小学/高中/四六级/考研。
- 是否做日语或韩语独立 App。
- 是否重构为跨平台共享核心。

决策依据：

- 留存。
- 付费转化。
- 内容生产成本。
- 用户反馈。
- 团队开发能力。
- 平台获客成本。

## 22. 验收标准

### 22.1 MVP 验收

必须满足：

- 首次安装后无需登录即可开始学习。
- 默认词库为中考必备。
- 可设置每日新词量。
- 今日新学和复习队列正确生成。
- 学习反馈后下次复习时间正确更新。
- 关闭 App 后学习记录不丢失。
- 无网络时仍可背已下载词库。
- 英美发音可切换。
- 收藏和笔记可保存。
- 词典查询可用。
- 拼写测试能记录正确/错误。
- 统计页能显示总进度。

### 22.2 内容验收

每个中考核心词必须有：

- 正确拼写。
- 至少一个中文核心释义。
- 词性。
- 发音。
- 至少一条例句。
- 内容来源记录。
- 人工审核状态。

### 22.3 体验验收

- 用户打开 App 后 2 次点击内开始学习。
- 每 10 个新词流程不超过 6-8 分钟。
- 单词页无明显遮挡和文字溢出。
- 设置项不会让用户困惑。
- 退出学习时不会丢失进度。

### 22.4 商用候选版本验收

商用候选版本除 MVP 验收外，还必须满足：

- 有隐私政策和用户协议。
- 有内容来源登记表。
- 有开源依赖许可证清单。
- 有崩溃监控和关键错误日志。
- 有用户反馈入口。
- 有内容纠错处理流程。
- 有内容包版本号和回滚方案。
- 有学习数据导出或云同步方案。
- 有支付失败、恢复购买、退款后的权益处理方案。
- 有至少一轮真机兼容测试。
- 有上架截图、应用介绍、权限说明。

### 22.5 数据指标验收

进入大规模推广前，应至少观察：

- 首次学习完成率。
- D1 和 D7 留存。
- 每日任务完成率。
- 平均学习时长。
- 复习逾期率。
- 拼写正确率变化。
- 内容纠错率。
- 崩溃率。
- 音频播放失败率。
- 付费入口点击率或付费意愿问卷结果。

## 23. 风险与对策

| 风险 | 等级 | 对策 |
| --- | --- | --- |
| 词典版权侵权 | 高 | 不内置未授权词典内容，建立来源登记 |
| 词库质量不稳定 | 高 | 人工审核中考核心词，建立纠错入口 |
| 复习量失控 | 高 | 加入复习上限和恢复计划 |
| 发音缺失 | 中 | TTS 兜底，本地缓存 |
| 功能过多导致延期 | 中 | MVP 严格聚焦中考必备 |
| 开源项目改造成本高 | 中 | 先做 3-5 天技术验证，不合适就自建 |
| 未成年人付费合规 | 高 | 商业化前完善协议和监护人机制 |
| 数据丢失 | 高 | Room 持久化、导出备份、迁移测试 |
| 用户留存不足 | 高 | 先做内测和公测，用数据决定是否扩功能 |
| 内容生产成本过高 | 高 | 首期聚焦中考核心词，建立审核流程后再扩词库 |
| 付费转化低 | 中 | 先验证权益点击和访谈，不急于复杂会员体系 |
| 学习流程过重 | 中 | 控制每日任务时长，展示预计完成时间 |
| 上架审核问题 | 中 | 提前准备权限说明、隐私政策、内容合规说明 |
| 支付和权益异常 | 高 | 商用前做订单、恢复权益、退款测试 |
| 埋点侵入隐私 | 高 | 只采集学习行为和技术指标，避免敏感内容 |
| 过早做多端多语种 | 高 | 英语 Android 数据未验证前不扩方向 |

## 24. 立即执行清单

### 24.1 产品决策

1. 确定首发产品名、包名和基础视觉方向。
2. 明确首期只做 Android 英语中考必备。
3. 确定是否二开 Flashcards Open Source App，或直接自建工程。
4. 确定 MVP 不做登录、不做会员、不做多语种、不做 iOS。
5. 确定内测用户来源：学生、家长、老师或自用测试。

### 24.2 技术任务

1. 建立 Android Kotlin + Compose 工程。
2. 定义 Room 数据库实体。
3. 建立 `core-domain`、`core-data`、`feature-study` 等基础分层。
4. 实现首页、单词页、学习计划页。
5. 实现 TTS 发音和英美音设置。
6. 接入开源 FSRS 调度。
7. 实现收藏、笔记、学习记录。
8. 加入基础埋点接口。
9. 加入崩溃日志或本地错误日志。
10. 建立数据库迁移测试。

### 24.3 内容任务

1. 准备 50 个中考词作为样例数据。
2. 整理中考必备词库 v1 数据来源。
3. 建立内容来源登记表。
4. 建立内容审核表。
5. 制定例句难度标准。
6. 制定音频生成和兜底策略。
7. 制定内容纠错处理流程。

### 24.4 商业准备

1. 设计首期关键指标看板。
2. 准备内测反馈表。
3. 起草隐私政策和用户协议草案。
4. 建立开源依赖许可证清单。
5. 记录未来可付费权益假设。
6. 准备 Phase 0 技术验证结论模板。
